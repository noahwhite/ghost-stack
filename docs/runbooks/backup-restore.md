# Backup and Restore Runbook

This runbook covers the nightly backup of Ghost stack data to Cloudflare R2, and how to restore from a backup if needed.

---

## Overview

### What is Backed Up

The `ghost-backup.service` syncs the entire `/var/mnt/storage/` directory on the Vultr block storage volume to R2 using `rclone sync`. This includes:

| Path | Contents |
|------|----------|
| `ghost/upload-data/` | Ghost content uploads (images, files) |
| `mysql/data/` | MySQL database files |
| `caddy/certs/` | Cloudflare origin certificates |
| `ghost-compose/` | Docker Compose generated config and `.env.secrets` |

### What is Excluded

The following paths are excluded from the backup:

| Excluded Path | Reason |
|---------------|--------|
| `ghost-compose/secrets/**` | Individual secret files — re-fetched from Infisical at boot; never backed up |
| `ghost-compose/.env.secrets` | Legacy secrets file — sourced from Infisical; never backed up |
| `ghost-compose/.env.generated` | Auto-generated by tinybird-provision.service at boot; not backed up |
| `sbin/**` | Operational scripts — deployed via Ignition; not backed up |

### Schedule

- **Frequency:** Daily
- **Randomized delay:** Up to 1 hour (prevents thundering herd)
- **Implemented by:** `ghost-backup.timer` (systemd timer) → `ghost-backup.service` (oneshot)
- **Estimated downtime:** ~2–3 minutes while ghost-compose is stopped

### Security Model

- R2 credentials are fetched from Infisical at boot by `infisical-secrets-fetch.service` and written to `/var/mnt/storage/ghost-compose/secrets/` (0600, root-only, persistent block storage)
- At backup runtime, credentials are written to `/run/rclone-backup.conf` (tmpfs, 0600) and shredded via EXIT trap after the backup completes — they exist only during the backup window
- R2 credentials are **never** passed as `-e` env vars to `docker run` (which would expose them in `docker inspect`)
- See [Token Rotation Runbook — R2 Backup Credentials](../token-rotation-runbook.md#r2-backup-credentials-ghost_dev_bckup_r2_access_key_id-and-ghost_dev_bckup_r2_secret_access_key) for rotation procedures

## Initial Provisioning

Before the first backup can run, you must manually create the R2 bucket, create a scoped API token, and add credentials to Infisical.

### Step 1: Create the R2 Bucket

1. Log into [Cloudflare Dashboard](https://dash.cloudflare.com)
2. Navigate to **Storage & Databases** → **R2 Object Storage** → **Overview**
3. Click **Create bucket**
4. Name: `ghost-backups-dev-separationofconcerns-dev`
5. Location: auto (or choose a region close to the Vultr dev instance)
6. Default Storage Class: **Standard**
7. Click **Create bucket**

### Step 2: Create a Scoped R2 API Token

1. In the Cloudflare Dashboard, navigate to **Storage & Databases** → **R2 Object Storage** → **Overview**
2. Click **Manage** in the **Account Details** section
3. Click **Create Account API token**
4. Name: `ghost-stack-backup-dev`
5. Permissions: **Object Read & Write** scoped to `ghost-backups-dev-separationofconcerns-dev` only
6. TTL: **90 days** (matching the R2 backup credential rotation period)
7. Click **Create API Token**
8. Cloudflare shows three values — save all three somewhere secure before leaving this page:
   - **Token value** — not used by this integration; store for your records
   - **Access Key ID** — store in Infisical as `GHOST_DEV_BCKUP_R2_ACCESS_KEY_ID`
   - **Secret Access Key** — store in Infisical as `GHOST_DEV_BCKUP_R2_SECRET_ACCESS_KEY`
9. Click **Finish**

### Step 3: Add R2 Credentials to Infisical

**Preferred — Infisical UI:**

1. Log into [app.infisical.com](https://app.infisical.com)
2. Navigate to the **Ghost Stack** project → **All Environments**
3. Click **Add Secret**: key `GHOST_DEV_BCKUP_R2_ACCESS_KEY_ID`, value = the Access Key ID from Step 2, Environments = Development
4. Click **Add Secret**: key `GHOST_DEV_BCKUP_R2_SECRET_ACCESS_KEY`, value = the Secret Access Key from Step 2, Environments = Development

**Alternative — CLI:**

```bash
# Authenticate if not already authenticated
infisical login \
  --method=universal-auth \
  --client-id="$INFISICAL_CLIENT_ID" \
  --client-secret="$INFISICAL_CLIENT_SECRET"

read -s SECRET_VALUE; export SECRET_VALUE
infisical secrets set GHOST_DEV_BCKUP_R2_ACCESS_KEY_ID="$SECRET_VALUE" \
  --projectId ghost-stack \
  --env dev

read -s SECRET_VALUE; export SECRET_VALUE
infisical secrets set GHOST_DEV_BCKUP_R2_SECRET_ACCESS_KEY="$SECRET_VALUE" \
  --projectId ghost-stack \
  --env dev

unset SECRET_VALUE
```

Verify both secrets are stored:

```bash
infisical secrets list --projectId ghost-stack --env dev
# Should now show GHOST_DEV_BCKUP_R2_ACCESS_KEY_ID and GHOST_DEV_BCKUP_R2_SECRET_ACCESS_KEY
```

### Step 4: Deploy Infrastructure

The R2 bucket name is the default value for `r2_dev_backups_bucket` in OpenTofu. The Cloudflare account ID is already available via `TF_VAR_cloudflare_account_id` (sourced from Bitwarden by `infra-shell.sh`). No additional Bitwarden or `infra-shell.sh` changes are needed.

After Infisical is provisioned, run `tofu plan` and `tofu apply` to deploy the instance with the backup service enabled.

```bash
tailscale ssh core@ghost-dev-01

sudo systemctl list-timers ghost-backup.timer
```

Expected output shows `ghost-backup.timer` with a future trigger time.

## Manually Triggering a Backup

```bash
tailscale ssh core@ghost-dev-01

sudo systemctl start ghost-backup.service
```

Follow the logs:

```bash
journalctl -u ghost-backup -f
```

**Expected log output:**

```
[ghost-backup] Stopping ghost-compose...
[ghost-backup] Running rclone sync to r2:ghost-backups-dev-separationofconcerns-dev...
[ghost-backup] Backup complete.
[ghost-backup] Restarting ghost-compose...
```

```bash
# View last backup run
journalctl -u ghost-backup --since "yesterday"

```

## Verifying Stack Recovery After Backup

After a backup completes (manually triggered or via timer), verify the Ghost stack recovered:

```bash
tailscale ssh core@ghost-dev-01

# All containers should be running
docker ps --format "table {{.Names}}\t{{.Status}}"
```

Expected: `caddy`, `ghost`, and `db` containers all show `Up`.

## Restore Procedure

Use this procedure to restore the Ghost stack from an R2 backup — for example, after provisioning a new instance following data loss.

### Prerequisites

- New instance has been provisioned via `tofu apply`
- Block storage is attached and mounted at `/var/mnt/storage/`
- SSH access via Tailscale is available

### Step 1: Run the restore script

A restore script is deployed to `/opt/bin/ghost-restore.sh` on the instance. It handles stopping ghost-compose, restoring from R2, and restarting ghost-compose — with an interactive confirmation prompt to prevent accidental execution.

```bash
tailscale ssh core@ghost-dev-01
sudo /opt/bin/ghost-restore.sh
```

The script will prompt: `Type 'yes' to continue:` — enter `yes` to proceed.

**Expected log output:**

```
[ghost-restore] Stopping ghost-compose...
[ghost-restore] Restoring from r2:ghost-backups-dev-separationofconcerns-dev to /var/mnt/storage...
[ghost-restore] Restore complete. Starting ghost-compose...
[ghost-restore] Done.
```

### Step 2: Verify containers started

```bash
docker ps --format "table {{.Names}}\t{{.Status}}"
```

### Step 4: Verify Ghost

```bash
# Check Ghost is responding
curl -sI https://separationofconcerns.dev
```

## Troubleshooting

### R2 secret files not found

**Symptom:** Backup exits with `ERROR: R2 secret files not found — was infisical-secrets-fetch.sh run at boot?`

**Diagnosis:** The secret files should have been written by `infisical-secrets-fetch.service` on first boot.

```bash
sudo ls -la /var/mnt/storage/ghost-compose/secrets/
journalctl -u infisical-secrets-fetch.service
```

**Resolution:**
- If the service ran but the files are missing, check if Infisical has the `GHOST_DEV_BCKUP_R2_ACCESS_KEY_ID` and `GHOST_DEV_BCKUP_R2_SECRET_ACCESS_KEY` secrets set
- If the token was already consumed (subsequent reboot), manually write the secret files using the rotation procedure in the [Token Rotation Runbook](../token-rotation-runbook.md#r2-backup-credentials-ghost_dev_bckup_r2_access_key_id-and-ghost_dev_bckup_r2_secret_access_key)

### Rclone authentication error

**Symptom:** Rclone exits with `AccessDenied` or `InvalidAccessKeyId`

**Diagnosis:** Credentials have expired or are incorrect.

**Resolution:** Follow the [R2 Backup Credentials rotation procedure](../token-rotation-runbook.md#r2-backup-credentials-ghost_dev_bckup_r2_access_key_id-and-ghost_dev_bckup_r2_secret_access_key).

### ghost-compose not recovering after backup

**Symptom:** Ghost containers are not running after a backup

**Diagnosis:** The EXIT trap should always restart ghost-compose. Check:

```bash
journalctl -u ghost-backup -n 50
journalctl -u ghost-compose -n 50
```

**Resolution:**
```bash
sudo systemctl start ghost-compose
```

### Rclone config not shredded

**Symptom:** `/run/rclone-backup.conf` still exists after backup

**Diagnosis:** This should not happen (EXIT trap runs on any exit). If it does:

```bash
sudo shred -u /run/rclone-backup.conf
```

Note: `/run/` is tmpfs on Flatcar — it is cleared on reboot regardless.
